## 네트워크 계층이 필요한 이유

데이터 링크 계층(스위치·MAC 주소)은 **동일 LAN 내** 통신만 담당합니다.  
인터넷처럼 **서로 다른 네트워크 간** 통신에는 **네트워크 계층(IP)** 이 필요합니다.

**데이터 링크 계층의 한계**
- 다른 네트워크로의 최적 경로 파악 불가
- 모든 네트워크 상 노드의 일괄 주소 지정 어려움

**네트워크 계층이 해결하는 것**
- IP 주소 지정 (Addressing)
- 라우팅 (Routing) — 최적 경로 결정
- 단편화 (Fragmentation) — MTU에 맞게 패킷 분할

---

## IP 주소 구조 (IPv4)

IPv4 주소는 **4바이트(32비트)**, 8비트씩 4개의 옥텟(octet)으로 구성됩니다.

```
192  .  168  .    1  .  100
 8bit    8bit    8bit    8bit
           = 32bit 총 = 2³² ≈ 42억 개
```

모든 IP 주소는 **네트워크 주소 + 호스트 주소** 두 부분으로 구성됩니다.

### 클래스 기반 주소 체계 (Classful)

| 클래스 | 시작 비트 | 네트워크/호스트 비트 | 범위 |
|--------|----------|-------------------|------|
| A | 0… | 8 / 24 | 0.x.x.x ~ 127.x.x.x |
| B | 10… | 16 / 16 | 128.x.x.x ~ 191.x.x.x |
| C | 110… | 24 / 8 | 192.x.x.x ~ 223.x.x.x |

> 호스트 주소가 모두 **0** → 네트워크 주소  
> 호스트 주소가 모두 **1** → 브로드캐스트 주소  
> 실제 할당 가능 호스트 = 2ⁿ - 2

## Classless Addressing — 서브넷 마스크 & CIDR

클래스 방식은 IP 낭비가 심해, 현재는 **CIDR(Classless Inter-Domain Routing)** 을 사용합니다.

### 서브넷 마스크 (Subnet Mask)

IP 주소에서 **네트워크 부분 = 1, 호스트 부분 = 0** 으로 표기한 비트열입니다.

```
IP:           192.168.1.100
              11000000.10101000.00000001.01100100

서브넷마스크:   255.255.255.0
              11111111.11111111.11111111.00000000
                                          ↑ 호스트 8비트 → 254개 호스트

AND 결과 =    192.168.1.0  ← 네트워크 주소
CIDR 표기:    192.168.1.0/24
```

`IP AND 서브넷마스크 = 네트워크 주소`

### CIDR 표기 (AWS에서 필수!)

```
10.0.0.0/8    → 255.0.0.0       → 호스트 16,777,214개
172.16.0.0/12 → 255.240.0.0     → 호스트 1,048,574개
192.168.0.0/16→ 255.255.0.0     → 호스트 65,534개
192.168.1.0/24→ 255.255.255.0   → 호스트 254개
10.0.1.0/28   → 255.255.255.240 → 호스트 14개
```

## 공인 IP vs 사설 IP

| 구분 | 설명 | 범위 |
|------|------|------|
| **공인 IP** (Public) | 인터넷에서 전 세계 유일한 주소 / ISP 할당 | 전 범위 |
| **사설 IP** (Private) | 내부 네트워크 전용. 라우터(공유기)가 할당 | 아래 3대역 |

**사설 IP 대역 (RFC 1918) — AWS VPC가 이 대역을 사용하는 이유!**

| 예약 주소 | IP 범위 | 사용 목적 |
|----------|---------|---------|
| 10.0.0.0/8 | 10.0.0.0 ~ 10.255.255.255 | 사설 네트워크 |
| 172.16.0.0/12 | 172.16.0.0 ~ 172.31.255.255 | 사설 네트워크 |
| 192.168.0.0/16 | 192.168.0.0 ~ 192.168.255.255 | 사설 네트워크 |
| 127.0.0.0/8 | 127.0.0.0 ~ 127.255.255.255 | 루프백(loopback) |

## NAT (Network Address Translation)

사설 IP를 가진 내부 호스트가 인터넷과 통신할 때,  
라우터가 **사설 IP ↔ 공인 IP** 를 변환합니다.

```
[내부 10.0.0.5:1234] → [NAT 라우터] → [인터넷 서버]
                          공인 IP: 203.0.113.1:45678 로 변환
                          NAT 변환 테이블에 기록
```

**NAPT (Network Address Port Translation)**:  
하나의 공인 IP + 포트 번호 조합으로 여러 내부 호스트를 동시에 인터넷에 연결합니다.  
가정의 공유기, 기업 네트워크, AWS NAT Gateway 모두 이 방식입니다.

**포트 포워딩**:  
외부에서 공인IP:특정포트 → 내부IP:포트 로 전달  
(예: 203.0.113.1:80 → 192.168.1.10:8080)

## IP 단편화 (Fragmentation)

전송하려는 패킷이 링크의 **MTU(Maximum Transmission Unit)** 보다 크면 분할해서 전송합니다.  
일반적인 MTU는 **1500 byte**입니다.

**IPv4 패킷 구조 中 단편화 관련 필드**

| 필드 | 설명 |
|------|------|
| **Identifier** | 원본 메시지 식별 (같은 값 = 같은 원본) |
| **Flag - DF** | Don't Fragment: 분할 금지 비트 |
| **Flag - MF** | More Fragment: 뒤에 조각 있음 |
| **Fragment Offset** | 원본에서 해당 조각의 위치 (8바이트 단위) |
| **TTL** | 라우터를 통과할 때마다 -1, 0이 되면 폐기 (무한 루프 방지) |

**Avoiding Fragmentation**: 모든 노드가 분할 없이 주고받을 수 있는 최대 크기를 미리 협상합니다.

## 라우팅 (Routing)

**라우팅**: 패킷이 이동할 **최적 경로를 결정**하는 과정  
**라우터**: 라우팅을 수행하는 장비 (공유기 = 소형 라우터)

### 라우팅 테이블

라우터가 참조하는 경로 정보표입니다. 각 항목은:

| 항목 | 의미 |
|------|------|
| 수신지 IP + 서브넷마스크 | 어디로 가는 패킷인가 |
| Next Hop (IP / Interface) | 다음으로 전달할 라우터 IP |
| Metric | 경로 비용 (낮을수록 선호) |

- **Default Route** (`0.0.0.0/0`): 일치하는 경로가 없을 때 사용. 보통 ISP 게이트웨이 방향

### 라우팅 프로토콜

라우터끼리 경로 정보를 교환하는 프로토콜입니다.

```
IGP (같은 AS 내부)
 ├── RIP   — 거리 벡터 방식, 최소 홉 수 계산, 주기적 업데이트 (최대 15홉)
 └── OSPF  — 링크 상태 방식, 네트워크 변경 시만 업데이트

EGP (AS 간)
 └── BGP   — 인터넷 백본 표준, 경로 정책 설정 가능
```

> **RIP**은 라우팅 테이블 갱신을 위해 주기적으로 라우터 간 통신  
> **OSPF**는 네트워크 구성 변경 시에만 라우팅 테이블 갱신

## ARP (Address Resolution Protocol)

IP 주소로 **MAC 주소를 알아내는** 프로토콜입니다.  
네트워크 계층 → 데이터 링크 계층 연결 역할을 합니다.

```
① ARP 요청 (Broadcast): "192.168.1.5의 MAC 주소 알려줘!"
② ARP 응답 (Unicast):   "저는 aa:bb:cc:dd:ee:ff입니다"
③ ARP 테이블 갱신:      IP ↔ MAC 매핑 저장 (일정 시간 TTL)
```

터미널에서 `arp -a` 명령어로 ARP 테이블을 확인할 수 있습니다.

## DHCP (Dynamic Host Configuration Protocol)

클라이언트에게 **IP 주소를 자동으로 할당**하는 프로토콜입니다.  
일반적으로 라우터(공유기)가 DHCP 서버 역할을 합니다.

```
클라이언트                         DHCP 서버
    │── ① DHCP Discover (Broadcast) ──→ │  "누가 IP 줄 수 있나요?"
    │←─ ② DHCP Offer ──────────────── │  "이 IP 드릴게요 (+ 임대기간, 마스크 등)"
    │── ③ DHCP Request (Broadcast) ──→ │  "그 IP 쓸게요!"
    │←─ ④ DHCP ACK ─────────────────  │  "확인, 사용하세요!"
```

**정적 IP 할당** vs **동적 IP 할당 (DHCP)**

| 구분 | 설명 | 사용 예 |
|------|------|---------|
| 정적 | 관리자가 직접 IP 지정 | 서버, 프린터 |
| 동적 | DHCP 서버가 자동 할당 | 일반 클라이언트 |

## AWS에서 네트워크 계층 개념 적용

| 개념 | AWS 서비스/기능 |
|------|---------------|
| 사설 IP 대역 | VPC CIDR (10.0.0.0/16 등) |
| 서브네팅 | VPC 서브넷 (퍼블릭 /24, 프라이빗 /24 등) |
| 라우터 | VPC Router (자동 생성) |
| 라우팅 테이블 | Route Table (서브넷별 경로 설정) |
| NAT | NAT Gateway (프라이빗 서브넷 → 인터넷) |
| DHCP | VPC DHCP Options Set |
| Default Route | 0.0.0.0/0 → Internet Gateway or NAT Gateway |

> **VPC 설계 예시**: 퍼블릭 서브넷(10.0.1.0/24)에 ALB·Bastion, 프라이빗 서브넷(10.0.2.0/24)에 EC2·RDS 배치  
> → 프라이빗 서브넷은 NAT Gateway를 통해 외부 통신

## 정리

- **IPv4** = 32비트, 네트워크 주소 + 호스트 주소로 구성
- **서브넷 마스크 & CIDR** — `/숫자`로 네트워크 크기 표현
- **사설 IP** (10.x, 172.16-31.x, 192.168.x) — AWS VPC의 기반
- **NAT** — 사설 IP를 공인 IP로 변환, AWS NAT Gateway와 동일 원리
- **DHCP** — IP 자동 할당, 라우터(공유기)가 수행
- **라우팅** — 최적 경로 결정, 라우팅 테이블 참조
