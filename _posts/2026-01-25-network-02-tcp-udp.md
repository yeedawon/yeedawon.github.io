## 전송 계층이란?

네트워크 계층(IP)이 **호스트 간** 통신을 담당한다면,  
전송 계층은 **호스트 내 특정 프로세스(포트)** 까지의 통신을 담당합니다.

```
전송 계층
 ├── TCP — 신뢰할 수 있는 통신 / 연결형 / 느림
 └── UDP — 신뢰할 수 없는 통신 / 비연결형 / 빠름
```

---

## 포트 (Port)

**포트**: 네트워크 상에서 특정 애플리케이션 프로세스를 식별하는 정보 (16비트, 0~65535)

```
특정 호스트의 특정 프로세스 = IP 주소 + 포트 번호
예: 192.168.1.5:8080  →  192.168.1.5 서버의 포트 8080번 프로세스
```

### 포트 번호 범위

| 구분 | 범위 | 설명 |
|------|------|------|
| 잘 알려진 포트 | 0 ~ 1023 | 표준 서비스 전용 |
| 등록된 포트 | 1024 ~ 49151 | 특정 애플리케이션 |
| 동적 포트 | 49152 ~ 65535 | 클라이언트 임시 포트 |

### 주요 포트 번호

| 포트 | 프로토콜 | 설명 |
|------|---------|------|
| 22 | SSH | 원격 접속 (AWS EC2 접속) |
| 80 | HTTP | 웹 |
| 443 | HTTPS | 보안 웹 |
| 53 | DNS | 도메인 이름 조회 |
| 3306 | MySQL | MySQL DB |
| 8080 | HTTP (대체) | Spring Boot 기본 포트 |

## TCP vs UDP

| 항목 | TCP | UDP |
|------|-----|-----|
| 연결 방식 | 연결형 (사전 연결 수립) | 비연결형 |
| 신뢰성 | 높음 (순서 보장, 재전송) | 낮음 |
| 속도 | 상대적으로 느림 | 빠름 |
| 헤더 크기 | 20~60 byte | 8 byte (고정) |
| 사용 예 | HTTP, HTTPS, FTP, SSH | DNS, DHCP, 스트리밍, 게임 |

**UDP 헤더** (4가지 필드만): 송신지 포트, 수신지 포트, 길이, 체크섬

## TCP 세그먼트 구조

```
 0          4          8         16                    32
 ┌──────────┬──────────┬──────────────────────────────┐
 │  송신지 포트          │  수신지 포트                    │
 ├──────────────────────────────────────────────────────┤
 │  순서 번호 (Sequence Number)                          │
 ├──────────────────────────────────────────────────────┤
 │  확인 응답 번호 (Acknowledgement Number)               │
 ├─────┬────┬──────────┬─────────────────────────────── ┤
 │데이터│예약 │ 제어 비트  │  윈도우                         │
 │오프셋│    │CWR·ECE·URG│ (수신 윈도우 크기)                │
 │     │    │ACK·PSH·RST│                               │
 │     │    │SYN·FIN    │                               │
 ├──────────────────────────────────────────────────────┤
 │  체크섬                │  긴급 포인터                    │
 └──────────────────────────────────────────────────────┘
```

**주요 제어 비트**

| 비트 | 의미 |
|------|------|
| **SYN** | 연결 수립 요청 |
| **ACK** | 세그먼트 수신 확인 |
| **FIN** | 연결 종료 요청 |

## TCP 연결 수립 — 3-way Handshake

```
호스트 A                        호스트 B
    │                               │
    │── SYN 세그먼트 ──────────────→ │  "연결 시작합니다"
    │                               │
    │← SYN+ACK 세그먼트 ─────────── │  "네, 확인했습니다. 연결 시작해요!"
    │                               │
    │── ACK 세그먼트 ──────────────→ │  "네, 확인했습니다"
    │                               │  (ESTABLISHED)
    │                               │
```

세 번의 메시지를 주고받은 후 **ESTABLISHED** 상태로 전환되며 데이터 송수신을 시작합니다.

---

## TCP 연결 종료 — 4-way Handshake

```
호스트 A (active close)          호스트 B (passive close)
    │── FIN 세그먼트 ────────────→ │  "연결 끊을게요"
    │← ACK 세그먼트 ─────────────  │  "확인"          ← B: CLOSE-WAIT
    │← FIN 세그먼트 ─────────────  │  "저도 끊을게요"
    │── ACK 세그먼트 ────────────→ │  "확인"          ← A: TIME-WAIT
```

A는 `TIME-WAIT` 상태에서 일정 시간 대기 후 완전히 종료합니다.  
(마지막 ACK 유실 시 B가 FIN을 재전송할 수 있도록)

---

## TCP 상태 (State)

TCP는 **Stateful 프로토콜**입니다. 연결의 현재 단계를 상태로 관리합니다.

```
CLOSED → (패시브 오픈) → LISTEN → (SYN 수신) → SYN-RECEIVED
                                                      ↓ ACK 수신
CLOSED ← (종료 완료) ← TIME-WAIT ←──────── ESTABLISHED
```

## 오류 제어 — ARQ (Automatic Repeat reQuest)

데이터가 제대로 전달되지 않으면 자동으로 재전송하는 메커니즘입니다.

### ① Stop-and-Wait ARQ

확인 응답(ACK)을 받기 전까지 다음 세그먼트를 보내지 않습니다.

```
A ─── seg1 ──→ B
A ←── ACK1 ─── B
A ─── seg2 ──→ B   ...
```

- 장점: 높은 신뢰성
- 단점: 네트워크 효율 낮음 (한 번에 하나씩)

### ② Go-Back-N ARQ (파이프라이닝)

여러 세그먼트를 연속 전송하다가 오류 발생 시,  
**오류 발생 세그먼트부터 이후 모든 것을 재전송**합니다.

```
A ─── 1,2,3,4,5 ──→ B
                      3번 손실!
A ←── NACK3 ──────── B
A ─── 3,4,5 ──────→ B  (3 이후 모두 재전송)
```

### ③ Selective Repeat ARQ

오류가 발생한 **특정 세그먼트만 재전송**합니다.  
각 패킷에 대한 개별 ACK를 발송합니다.

## 흐름 제어 — 슬라이딩 윈도우

수신 호스트가 한 번에 받을 수 있는 데이터 양을 **수신 윈도우(rwnd)** 로 송신 호스트에 알려줍니다.  
송신 호스트는 확인 응답 없이 **윈도우 크기만큼** 연속으로 보낼 수 있습니다.

```
윈도우 크기 = 5인 경우:
A ─── [1][2][3][4][5] ──→ B  (ACK 없이 5개 연속 전송)
A ←── ACK 3 ─────────── B  (윈도우 슬라이드)
A ─── [4][5][6][7][8] ──→ B
```

## 혼잡 제어

네트워크 혼잡(패킷 손실·지연)을 감지하고 **송신량을 줄이는** 메커니즘입니다.

### 혼잡 신호

1. **ACK 중복 수신** (3번 연속) → 빠른 재전송 (Fast Retransmit)
2. **타임아웃** → 심각한 혼잡

### 혼잡 제어 알고리즘

**AIMD (Additive Increase / Multiplicative Decrease)**

```
혼잡 없을 때: RTT마다 혼잡 윈도우를 1씩 선형 증가
혼잡 감지 시: 혼잡 윈도우를 1/2로 감소
```

**슬로 스타트 (Slow Start)**

```
초기: 혼잡 윈도우 = 1 MSS
     ACK를 받을 때마다 +1 MSS → RTT마다 2배씩 지수적 증가
임계점(ssthresh)에 도달하면 AIMD 방식으로 전환
```

**혼잡 제어 전환 규칙**

| 상황 | 조치 |
|------|------|
| 슬로 스타트 임계점 도달 | 선형 증가로 전환 |
| 타임아웃 | 혼잡 윈도우 = 1, ssthresh = 현재/2, 슬로 스타트 재시작 |
| ACK 3중 중복 | ssthresh = 현재/2, 빠른 복구 |

### ECN (Explicit Congestion Notification)

라우터가 혼잡 상태를 IP 패킷 헤더의 **ECN 비트**로 직접 알려줍니다.  
타임아웃 이전에 사전 방지가 가능합니다.

```
호스트A → (패킷) → 라우터: 혼잡 감지! ECN 비트 = 11
라우터 → (패킷) → 호스트B: "ECN이 1이네? 라우터가 혼잡하구나"
호스트B → TCP 세그먼트 ECE 비트 = 1 → 호스트A
호스트A → TCP 세그먼트 CWR 비트 = 1 → 혼잡 윈도우 조절
```

## MSS (Maximum Segment Size)

TCP로 전송할 수 있는 **최대 페이로드 크기** (TCP 헤더 제외)입니다.

```
IP MTU (1500 byte)
│
├── IP 헤더 (20 byte)
│   └── TCP 헤더 (20 byte)
│       └── 페이로드 (1460 byte) ← MSS
```

## 백엔드 / AWS에서의 TCP

- **Spring Boot의 HTTP 요청**: HTTP/1.1 = TCP 기반. keep-alive로 한 연결에서 여러 요청 처리
- **AWS EC2 보안 그룹**: 포트(22/SSH, 8080/Spring Boot, 3306/MySQL)별 인바운드/아웃바운드 규칙
- **AWS ELB**: 클라이언트 ↔ 로드밸런서 ↔ EC2 각 구간에 독립된 TCP 연결
- **RDS**: MySQL(3306)을 통한 JDBC 연결도 TCP 기반

## 정리

- **포트**: 호스트 내 프로세스를 식별 (IP+포트 = 소켓)
- **TCP**: 연결형, 신뢰성, 오류·흐름·혼잡 제어 내장
- **UDP**: 비연결형, 빠르고 단순, DNS·실시간 스트리밍
- **3-way Handshake**: SYN → SYN+ACK → ACK
- **슬라이딩 윈도우**: 흐름 제어, ACK 전에 여러 세그먼트 전송
- **슬로 스타트 + AIMD**: 혼잡 제어 기본 알고리즘
- 다음 포스트: **응용 계층 — HTTP, DNS, URI**
